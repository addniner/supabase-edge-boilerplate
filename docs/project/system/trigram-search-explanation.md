# Trigram 검색 원리 상세 설명

## 1. Trigram이란?

**Trigram**은 문자열을 **3글자 단위**로 쪼개는 방법입니다.

### 예시: "사과사진"

```
"사과사진"을 Trigram으로 분해:
- (빈칸)사과  ← 앞에 빈칸 추가
- 사과사
- 과사진
- 사진(빈칸)  ← 뒤에 빈칸 추가
```

### 예시: "사과 사진" (띄어쓰기 포함)

```
"사과 사진"을 Trigram으로 분해:
- (빈칸)사과
- 사과(공백)
- (공백)사진
- 사진(빈칸)
```

## 2. 유사도 계산 (Similarity)

PostgreSQL의 `pg_trgm` 확장은 두 문자열의 Trigram을 비교하여 **유사도 점수**를 계산합니다.

### 유사도 공식

```
similarity(A, B) = 공통 Trigram 개수 / 전체 고유 Trigram 개수
```

### 실제 계산 예시

**"사과사진" vs "사과 사진"**

```
"사과사진"의 Trigram: {(빈칸)사과, 사과사, 과사진, 사진(빈칸)}
"사과 사진"의 Trigram: {(빈칸)사과, 사과(공백), (공백)사진, 사진(빈칸)}

공통 Trigram:
- (빈칸)사과 ✅
- 사진(빈칸) ✅

전체 고유 Trigram: 6개
공통 Trigram: 2개

유사도 = 2/6 = 0.33 (약 33%)
```

## 3. 연산자 동작 원리

### `%` 연산자 (Similarity Operator)

```sql
WHERE file_name % '사과사진'
```

이 쿼리는:
1. `file_name`의 각 값을 Trigram으로 분해
2. '사과사진'을 Trigram으로 분해
3. 유사도를 계산
4. **유사도가 임계값(threshold) 이상이면 매칭**

**기본 임계값**: `pg_trgm.similarity_threshold = 0.3` (30%)

### `<->` 연산자 (Distance Operator)

```sql
ORDER BY file_name <-> '사과사진'
```

이 연산자는:
- **거리(거리가 가까울수록 유사함)**를 반환
- 거리 = `1 - similarity`
- 예: 유사도 0.33 → 거리 0.67

**정렬 효과**:
- 거리가 작을수록(유사도가 높을수록) 먼저 정렬
- "사과 사진"이 "사과가 썪은 사진"보다 먼저 나옴

## 4. GIN 인덱스의 역할

### 인덱스 구조

```
GIN (Generalized Inverted Index) 인덱스:

Trigram → 파일명 목록
(빈칸)사과 → ["사과사진", "사과 사진", "사과.jpg", ...]
사과사 → ["사과사진", "사과사과", ...]
과사진 → ["사과사진", "과사진.jpg", ...]
...
```

### 검색 과정

1. **검색어 "사과사진"을 Trigram으로 분해**
   ```
   {(빈칸)사과, 사과사, 과사진, 사진(빈칸)}
   ```

2. **인덱스에서 각 Trigram이 포함된 파일명 찾기**
   ```
   (빈칸)사과 → ["사과사진", "사과 사진", ...]
   사과사 → ["사과사진", ...]
   과사진 → ["사과사진", ...]
   사진(빈칸) → ["사과사진", "사과 사진", ...]
   ```

3. **교집합(Intersection) 찾기**
   ```
   모든 Trigram에 공통으로 포함된 파일명 = 후보
   → ["사과사진", "사과 사진"]
   ```

4. **유사도 계산 및 필터링**
   ```
   각 후보에 대해 유사도 계산
   - "사과사진": 유사도 1.0 (완전 일치)
   - "사과 사진": 유사도 0.33
   
   임계값 0.3 이상만 반환
   → 둘 다 반환됨 ✅
   ```

## 5. 왜 빠른가?

### 기존 LIKE 방식의 문제

```sql
-- ❌ 느린 방식
WHERE file_name ILIKE '%사과사진%'
```

**문제점**:
- 앞뒤 와일드카드(`%`) 때문에 **인덱스를 사용할 수 없음**
- **전체 테이블 스캔(Full Table Scan)** 필요
- 데이터가 많아질수록 느려짐

### Trigram + GIN 인덱스의 장점

```sql
-- ✅ 빠른 방식
WHERE file_name % '사과사진'
```

**장점**:
- GIN 인덱스를 활용하여 **빠른 검색**
- 데이터가 많아져도 성능 유지
- 띄어쓰기 무시 검색 가능

## 6. 실제 동작 예시

### 케이스 1: 정확한 일치

```
검색어: "사과사진"
파일명: "사과사진.jpg"

Trigram 비교:
- "사과사진": {(빈칸)사과, 사과사, 과사진, 사진(빈칸)}
- "사과사진": {(빈칸)사과, 사과사, 과사진, 사진(빈칸)}

유사도: 4/4 = 1.0 (100%)
거리: 0
→ 최상위 정렬 ✅
```

### 케이스 2: 띄어쓰기 차이

```
검색어: "사과사진"
파일명: "사과 사진.jpg"

Trigram 비교:
- "사과사진": {(빈칸)사과, 사과사, 과사진, 사진(빈칸)}
- "사과 사진": {(빈칸)사과, 사과(공백), (공백)사진, 사진(빈칸)}

공통: {(빈칸)사과, 사진(빈칸)}
유사도: 2/6 = 0.33 (33%)
거리: 0.67
→ 임계값(0.3) 이상이므로 검색됨 ✅
→ 정확한 일치보다는 뒤에 정렬됨
```

### 케이스 3: 부분 일치

```
검색어: "사과"
파일명: "사과 사진.jpg"

Trigram 비교:
- "사과": {(빈칸)사과, 사과(빈칸)}
- "사과 사진": {(빈칸)사과, 사과(공백), (공백)사진, 사진(빈칸)}

공통: {(빈칸)사과}
유사도: 1/4 = 0.25 (25%)
거리: 0.75
→ 임계값(0.3) 미만이므로 검색 안됨 ❌
```

**해결책**: 검색어가 너무 짧으면(1글자) ILIKE 사용

## 7. 코드에서의 구현

### 검색어 길이별 처리

```typescript
// 1글자: ILIKE 사용 (Trigram 비효율)
if (trimmed.length < 2) {
  return {
    applyCondition: (qb) => qb.where(sql`file_name ILIKE ${`%${trimmed}%`}`),
    useDistance: false,
  };
}

// 2글자 이상: Trigram 사용
return {
  applyCondition: (qb) => qb.where(sql`file_name % ${trimmed}`),
  useDistance: true, // 거리 연산자로 정렬 가능
};
```

### 정렬 로직

```typescript
if (searchCondition?.useDistance) {
  query = query
    .orderBy(sql`file_name <-> ${searchTerm}`, "asc") // 유사도 순
    .orderBy("createdAt", "desc") // 동일 유사도면 최신순
    .orderBy("id", "desc"); // 동일 시간이면 ID 순
}
```

## 9. 주의사항

### 1. 인덱스 용량

- GIN 인덱스는 일반 B-Tree 인덱스보다 큼
- 파일명이 길수록 인덱스 크기 증가
- 예: 10만 건 기준 약 50-100MB

### 2. 쓰기 성능

- INSERT/UPDATE 시 Trigram 생성 필요
- 약간의 쓰기 성능 저하 (보통 무시 가능한 수준)

### 3. 임계값 조정

```sql
-- 기본값: 0.3
SET pg_trgm.similarity_threshold = 0.4; -- 더 엄격하게
SET pg_trgm.similarity_threshold = 0.2; -- 더 관대하게
```

- 너무 낮으면: 엉뚱한 결과 포함
- 너무 높으면: 띄어쓰기 무시가 안될 수 있음

## 10. 요약

1. **Trigram**: 문자열을 3글자 단위로 분해
2. **유사도**: 공통 Trigram 비율로 계산
3. **GIN 인덱스**: 빠른 검색을 위한 역인덱스 구조
4. **% 연산자**: 유사도 기반 매칭
5. **<-> 연산자**: 거리 기반 정렬
6. **장점**: 띄어쓰기 무시, 빠른 성능, 인덱스 활용
7. **단점**: 인덱스 용량 증가, 짧은 검색어 비효율
